## 1. プロジェクト概要

### 1.1 コンセプト

サーマルレシートプリンターを使い、白黒のレトロな写真を印刷する**プリクラサービス**。
スマホで4枚連続撮影 → フィルター適用 → 2x2コラージュ生成 → レシートプリンターから印刷、
というデジタルとアナログが融合した体験を提供する。

ハッカソン出展作品として、**あえてオーバーアーキテクチャなAWS構成**で構築し、技術的インパクトを狙う。
「レシートにプリクラを印刷するためだけに、ここまでやるのか」という驚きが審査のキーポイント。

### 1.2 プロジェクト名

**Receipt Purikura**（レシートプリクラ）

### 1.3 ターゲット

- ハッカソン来場者・審査員
- 技術に興味がある人

### 1.4 成功指標

- デモで来場者が「面白い」と反応すること
- 撮影〜印刷まで**30秒以内**で完了すること
- 50人以上の同時アクセスに耐えること
- AWS構成の壮大さで審査員にインパクトを与えること

## 2. ユーザー体験フロー

```
[来場者]                       [MacBook]                [AWS]
   |                             |                       |
   |  1. QRコード表示中           |                       |
   |<---- QRスキャン ────────────|                       |
   |                             |                       |
   |  2. Webアプリが開く          |                       |
   |  3. フィルター選択           |                       |
   |  4. カメラ起動               |                       |
   |  5. 📸 1枚目撮影            |                       |
   |  6. 📸 2枚目撮影            |                       |
   |  7. 📸 3枚目撮影            |                       |
   |  8. 📸 4枚目撮影            |                       |
   |  9. 4枚アップロード ─────────+────→ S3 Presigned     |
   |                             |        URL Upload      |
   | 10. 「処理中...」表示        |                       |
   |     (WebSocket通知) <───────+────── Step Functions   |
   |                             |       Pipeline         |
   | 11. コラージュ結果表示       |                       |
   |                             |                       |
   |                             | 12. SQS → 印刷ジョブ   |
   |                             |     ポーリング         |
   |                             | 13. レシート印刷       |
   |                             |                       |
   | 14. レシートのQRスキャン      |                       |
   |     → カラー版DL             |                       |
```

### 詳細ステップ

| # | 操作 | 説明 |
|---|------|------|
| 1 | QRコード表示 | MacBook画面にWebアプリのURLをQRコードで表示 |
| 2 | QRスキャン | 来場者がスマホでスキャン → ブラウザでWebアプリが開く |
| 3 | フィルター選択 | フィルター（美肌/モノクロ/セピア等）を選択 |
| 4 | カメラ起動 | スマホのフロントカメラが起動（許可ダイアログ対応） |
| 5-8 | 4枚連続撮影 | カウントダウン付きで4枚を連続撮影（各3秒間隔） |
| 9 | アップロード | 4枚をまとめてS3 Presigned URLで直接アップロード |
| 10 | 処理中表示 | WebSocketでリアルタイムに進捗を表示 |
| 11 | 結果表示 | スマホにカラー版の2x2コラージュを表示 |
| 12 | 印刷ジョブ | SQSキューに印刷ジョブが入る |
| 13 | レシート印刷 | MacBookがSQSをポーリング → プリンターで印刷（自動カット） |
| 14 | カラー版DL | レシートのQRコードをスキャン → カラー版コラージュをダウンロード |

## 3. 機能要件

### 3.1 フロントエンド（スマホWebアプリ）

| ID | 機能 | 優先度 | 説明 |
|----|------|--------|------|
| F-01 | QRコード起動 | Must | QRスキャンでWebアプリが開く |
| F-02 | フィルター選択 | Must | 撮影前にフィルターを選択できる |
| F-03 | カメラ撮影（4枚連続） | Must | カウントダウン付きで4枚を連続撮影できる |
| F-04 | 撮り直し | Should | 撮影後に特定の写真を撮り直せる |
| F-05 | 画像アップロード | Must | 撮影した4枚をS3にアップロードできる |
| F-06 | 処理状態表示 | Must | 処理中→完了の状態をリアルタイムで表示 |
| F-07 | コラージュ結果表示 | Must | カラー版の2x2コラージュを表示できる |
| F-08 | カラー版ダウンロード | Should | QRコード経由でカラー版を再度ダウンロードできる |

### 3.2 フィルター処理

2種類のフィルターカテゴリを提供する:
- **簡易フィルター（Pillow）**: 即座に適用。美肌・モノクロ等の定番フィルター
- **AIスタイル変換（Stability AI Style Transfer on Bedrock）**: 10-20秒かかるが、写真の構図を保ったままAIが画風を変換する高品質フィルター

#### 簡易フィルター（Lambda + Pillow）

| ID | フィルター | 優先度 | 処理時間 | 説明 |
|----|-----------|--------|---------|------|
| FL-01 | ナチュラル | Must | 即座 | フィルターなし（元写真のまま） |
| FL-02 | 美肌 | Must | ~1秒 | 肌をなめらかに補正（ガウシアンぼかし + ブレンド） |
| FL-03 | 明るさ補正 | Must | ~1秒 | 写真を明るく、映える仕上がりに |
| FL-04 | モノクロ | Should | ~1秒 | 白黒フィルター（レシート印刷との相性◎） |
| FL-05 | セピア | Should | ~1秒 | レトロな色味に変換 |

#### AIスタイル変換（Bedrock Stability AI Style Transfer）

| ID | スタイル | 優先度 | 処理時間 | 説明 |
|----|---------|--------|---------|------|
| AI-01 | アニメ風 | Should | ~10-20秒 | アニメ・イラスト風に変換 |
| AI-02 | ポップアート風 | Should | ~10-20秒 | ポップアートのカラフルな画風に変換 |
| AI-03 | 水彩画風 | Nice | ~10-20秒 | 水彩画のような柔らかいタッチに変換 |

> **AIスタイル変換の仕組み**: Stability AI Style Transfer (`us.stability.stable-style-transfer-v1:0`) を使用。
> 事前に用意した「参照スタイル画像」と撮影した写真を入力すると、
> 写真の構図・人物はそのままに画風だけをAIが変換する。
> `style_strength`, `composition_fidelity`, `change_strength` パラメータで変換の度合いを調整可能。
>
> **コスト**: ~$0.03-0.06/枚 × 4枚 = ~$0.12-0.24/セッション
>
> **注意**: AIフィルター選択時は処理時間が長くなることをユーザーに事前に表示する。

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目 | 要件 |
|------|------|
| 4枚撮影〜印刷完了（簡易フィルター） | 20秒以内（撮影時間除く） |
| 4枚撮影〜印刷完了（AIスタイル） | 40秒以内（撮影時間除く） |
| 画像アップロード（4枚） | 5秒以内（WiFi環境） |
| 画像処理（簡易フィルター） | 5秒以内 |
| 画像処理（AIスタイル変換） | 20秒以内（4枚並列） |
| 印刷所要時間 | 10秒以内 |
| 同時接続ユーザー | 50人以上 |

### 4.2 可用性

| 項目 | 要件 |
|------|------|
| サービス稼働時間 | イベント開催中は100%（数時間の短期運用） |
| プリンター障害 | エラー通知 + ユーザーへのフォールバック表示 |

### 4.3 セキュリティ

| 項目 | 要件 |
|------|------|
| 認証 | なし（匿名利用） |
| 画像保存 | イベント終了後に一括削除可能にする（TTL設定） |
| Presigned URL | 有効期限を短く設定（5分） |
| HTTPS | 全通信をHTTPS化（カメラAPI利用に必須） |

### 4.4 対応ブラウザ・端末

| 項目 | 要件 |
|------|------|
| iOS Safari | 必須対応 |
| Android Chrome | 必須対応 |
| デスクトップ | QRコード表示・ダッシュボード用のみ |

## 9. 画像処理パイプライン詳細（Step Functions）

### 9.1 処理フロー

```
[4枚アップロード完了 → 処理開始トリガー]
        │
        ▼
┌──────────────┐
│ Step 1       │  Amazon Rekognition
│ 顔検出       │  → 4枚それぞれの顔座標を取得
│ (並列処理)   │  → 顔なしでも続行
└──────┬───────┘
       │
       ▼
       ┌───── フィルター種別で分岐 ─────┐
       │                               │
  [簡易フィルター]              [AIスタイル変換]
       │                               │
       ▼                               ▼
┌──────────────┐              ┌──────────────────┐
│ Step 2a      │              │ Step 2b           │
│ Pillow       │              │ Stability AI      │
│ フィルター   │              │ Style Transfer    │
│ 適用(並列)   │              │ (4枚並列でAPI呼出) │
│ ~1秒         │              │ ~10-20秒          │
└──────┬───────┘              └────────┬─────────┘
       │                               │
       └──────────┬────────────────────┘
                  │
                  ▼
┌──────────────┐
│ Step 3       │  Lambda + Pillow
│ コラージュ   │  → 4枚を2x2グリッドに配置
│ 生成         │  → フレーム・装飾をオーバーレイ
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Step 4       │  Amazon Bedrock (Claude)
│ キャプション  │  → コラージュ画像を分析
│ 生成         │  → 面白いキャプションを生成
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Step 5       │  Lambda + Pillow
│ 印刷用画像   │  → ディザリング（白黒2値化）
│ 生成         │  → QRコード埋め込み
│              │  → レシートレイアウト生成
└──────┬───────┘
       │
       ├──→ S3に保存（カラー版コラージュ + 印刷用画像）
       ├──→ DynamoDBにメタデータ書き込み
       ├──→ WebSocketでクライアントに完了通知
       └──→ SQSに印刷ジョブ送信
```

### 9.2 フィルター処理の実装方針

#### 簡易フィルター（Lambda + Pillow）

| フィルター | Pillow実装 | 処理時間 |
|-----------|-----------|---------|
| ナチュラル | 処理なし（元画像のまま） | 即座 |
| 美肌 | `GaussianBlur` で平滑化 → 元画像と `ImageChops.blend` でブレンド | ~1秒 |
| 明るさ補正 | `ImageEnhance.Brightness` + `ImageEnhance.Contrast` | ~1秒 |
| モノクロ | `ImageOps.grayscale` | ~1秒 |
| セピア | グレースケール化 → セピアカラーマップ適用 | ~1秒 |

#### AIスタイル変換（Bedrock Stability AI Style Transfer）

| スタイル | 参照画像 | パラメータ目安 |
|---------|---------|--------------|
| アニメ風 | アニメ調のイラスト参照画像を事前用意 | `style_strength: 0.6`, `composition_fidelity: 0.8` |
| ポップアート風 | ポップアートの参照画像を事前用意 | `style_strength: 0.7`, `composition_fidelity: 0.7` |
| 水彩画風 | 水彩画の参照画像を事前用意 | `style_strength: 0.5`, `composition_fidelity: 0.8` |

**処理フロー:**
1. 4枚の写真に対してBedrock APIを**並列呼び出し**
2. 各写真 + 参照スタイル画像 → Style Transfer API → 変換済み画像
3. 並列処理により、4枚でも壁時計時間は1枚分（~10-20秒）

**コスト**: ~$0.03-0.06/枚 × 4枚 = ~$0.12-0.24/セッション

### 9.3 コラージュ生成の仕様

```
┌─────────────────────────┐
│  padding: 10px          │
│  ┌──────────┬──────────┐ │
│  │          │          │ │
│  │  写真1    │  写真2   │ │  各写真: 正方形にクロップ
│  │ (顔中心) │ (顔中心) │ │  （顔検出結果を使用）
│  │          │          │ │
│  ├──────────┼──────────┤ │
│  │          │          │ │
│  │  写真3    │  写真4   │ │
│  │ (顔中心) │ (顔中心) │ │
│  │          │          │ │
│  └──────────┴──────────┘ │
│  gap: 6px               │
└─────────────────────────┘
```

- 各写真は正方形にクロップ（顔検出で顔を中心に配置）
- 写真間のgap: 6px
- 外側のpadding: 10px
- 出力サイズ: 576x576px（レシート80mm幅に最適化）

### 9.4 ディザリングアルゴリズム

| アルゴリズム | 特徴 | 推奨度 |
|-------------|------|--------|
| Floyd-Steinberg | 誤差拡散法。自然な階調表現。最も一般的 | ★★★ |
| Atkinson | Mac初期のディザリング。レトロ感が強い | ★★★ |
| Ordered (Bayer) | 規則的なパターン | ★★ |

> 事前検証で最適なアルゴリズムを選定する。

## 13. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| AIスタイル変換の品質が不安定 | 中 | 参照スタイル画像とパラメータを事前に十分チューニング。最悪簡易フィルターのみで運用 |
| AIスタイル変換のレイテンシ | 中 | 4枚並列処理で壁時計時間を短縮。処理中UIで待ち時間を演出 |
| Stability AI Style Transferのコスト | 低 | ~$0.24/セッション。ハッカソン期間中は許容範囲 |
| プリンターの画質が粗い | 中 | ディザリングアルゴリズムの事前検証で最適化 |
| 4枚撮影のUXが悪い | 中 | カウントダウンUI・撮り直し機能で体験を向上 |
| **iOS Safariのカメラ挙動** | **高** | getUserMedia の機種差が大きい。**Day 2-3で実機検証必須** |
| **WebSocket構築の工数オーバー** | **高** | 詰まったら即ポーリング（5秒間隔GET）に切り替える |
| **IaC構築の工数オーバー** | **高** | マネジメントコンソール手動構築で進め、余裕があればコード化 |
| 会場WiFiの速度不足 | 中 | 画像を圧縮してアップロード。モバイルテザリングも準備 |
| Lambda コールドスタート | 低 | Provisioned Concurrencyの設定 |
| 50人同時接続時の遅延 | 中 | SQS + Step Functionsで非同期処理。印刷は直列だが処理は並列 |
| プリンター紙切れ | 低 | 予備ロール紙を十分に準備 |
| 14日間で全機能の実装が間に合わない | 高 | 優先度（Must > Should > Nice）に従い段階的に実装 |
| 4枚アップロードの通信量 | 中 | 画像をリサイズ・圧縮（最大1280px、JPEG品質80%） |

## 17. 用語集

| 用語 | 説明 |
|------|------|
| ディザリング | カラーやグレースケールの画像を白黒2値（白/黒のドットのみ）に変換する技術。ドットの密度で濃淡を表現する |
| コラージュ | 複数の写真を1枚の画像にレイアウトして合成すること。本プロジェクトでは4枚を2x2グリッドに配置する |
| ESC/POS | レシートプリンターの標準制御コマンド体系。ほぼ全てのレシートプリンターが対応 |
| Presigned URL | 一時的にS3への直接アップロードを許可する署名付きURL。サーバーを経由せず直接S3にアップロードできる |
| Floyd-Steinberg | 最も一般的なディザリングアルゴリズム。誤差を周囲のピクセルに拡散させて自然な階調を表現する |
| WebSocket | サーバーからクライアントにリアルタイムでデータを送信できる双方向通信プロトコル |
| Step Functions | AWSのワークフローオーケストレーションサービス。複数のLambda関数を順番/並列に実行する |
| SQS | AWSのメッセージキューサービス。非同期でジョブを受け渡す仕組み |
| Pillow | Pythonの画像処理ライブラリ。フィルター適用、リサイズ、合成などの画像操作ができる |
