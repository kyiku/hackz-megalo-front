## 1. プロジェクト概要

### 1.1 コンセプト

サーマルレシートプリンターを使い、白黒のレトロな写真を印刷する**プリクラサービス**。
スマホで4枚連続撮影 → フィルター適用 → 2x2コラージュ生成 → レシートプリンターから印刷、
というデジタルとアナログが融合した体験を提供する。

ハッカソン出展作品として、**あえてオーバーアーキテクチャなAWS構成**で構築し、技術的インパクトを狙う。
「レシートにプリクラを印刷するためだけに、ここまでやるのか」という驚きが審査のキーポイント。

### 1.2 プロジェクト名

**Receipt Purikura**（レシートプリクラ）

### 1.3 ターゲット

- ハッカソン来場者・審査員
- 技術に興味がある人

### 1.4 成功指標

- デモで来場者が「面白い」と反応すること
- 撮影〜印刷まで**30秒以内**で完了すること
- 50人以上の同時アクセスに耐えること
- AWS構成の壮大さで審査員にインパクトを与えること

## 2. ユーザー体験フロー

```
[来場者]                       [MacBook]                [AWS]
   |                             |                       |
   |  1. QRコード表示中           |                       |
   |<---- QRスキャン ────────────|                       |
   |                             |                       |
   |  2. Webアプリが開く          |                       |
   |  3. フィルター選択           |                       |
   |  4. カメラ起動               |                       |
   |  5. 📸 1枚目撮影            |                       |
   |  6. 📸 2枚目撮影            |                       |
   |  7. 📸 3枚目撮影            |                       |
   |  8. 📸 4枚目撮影            |                       |
   |  9. 4枚アップロード ─────────+────→ S3 Presigned     |
   |                             |        URL Upload      |
   | 10. 「処理中...」表示        |                       |
   |     (WebSocket通知) <───────+────── Step Functions   |
   |                             |       Pipeline         |
   | 11. コラージュ結果表示       |                       |
   |                             |                       |
   |                             | 12. IoT Core MQTT      |
   |                             |     → 印刷ジョブ送信   |
   |                             | 13. レシート印刷       |
   |                             |                       |
   | 14. レシートのQRスキャン      |                       |
   |     → カラー版DL             |                       |
```

### 詳細ステップ

| # | 操作 | 説明 |
|---|------|------|
| 1 | QRコード表示 | MacBook画面にWebアプリのURLをQRコードで表示 |
| 2 | QRスキャン | 来場者がスマホでスキャン → ブラウザでWebアプリが開く |
| 3 | フィルター選択 | フィルター（美肌/モノクロ/セピア等）を選択 |
| 4 | カメラ起動 | スマホのフロントカメラが起動（許可ダイアログ対応） |
| 5-8 | 4枚連続撮影 | カウントダウン付きで4枚を連続撮影（各3秒間隔） |
| 9 | アップロード | 4枚をまとめてS3 Presigned URLで直接アップロード |
| 10 | 処理中表示 | WebSocketでリアルタイムに進捗を表示 |
| 11 | 結果表示 | スマホにカラー版の2x2コラージュを表示 |
| 12 | 印刷ジョブ | IoT Core MQTT経由で印刷ジョブが送信される |
| 13 | レシート印刷 | MacBookがMQTTで受信 → プリンターで印刷（自動カット） |
| 14 | カラー版DL | レシートのQRコードをスキャン → カラー版コラージュをダウンロード |

### 3.2 フィルター処理

2種類のフィルターカテゴリを提供する:
- **簡易フィルター（sharp）**: 即座に適用。美肌・モノクロ等の定番フィルター
- **AIスタイル変換（Stability AI Style Transfer on Bedrock）**: 10-20秒かかるが、写真の構図を保ったままAIが画風を変換する高品質フィルター

#### 簡易フィルター（Lambda + sharp）

| ID | フィルター | 優先度 | 処理時間 | 説明 |
|----|-----------|--------|---------|------|
| FL-01 | ナチュラル | Must | 即座 | フィルターなし（元写真のまま） |
| FL-02 | 美肌 | Must | ~1秒 | 肌をなめらかに補正（ガウシアンぼかし + ブレンド） |
| FL-03 | 明るさ補正 | Must | ~1秒 | 写真を明るく、映える仕上がりに |
| FL-04 | モノクロ | Should | ~1秒 | 白黒フィルター（レシート印刷との相性◎） |
| FL-05 | セピア | Should | ~1秒 | レトロな色味に変換 |

#### AIスタイル変換（Bedrock Stability AI Style Transfer）

| ID | スタイル | 優先度 | 処理時間 | 説明 |
|----|---------|--------|---------|------|
| AI-01 | アニメ風 | Should | ~10-20秒 | アニメ・イラスト風に変換 |
| AI-02 | ポップアート風 | Should | ~10-20秒 | ポップアートのカラフルな画風に変換 |
| AI-03 | 水彩画風 | Nice | ~10-20秒 | 水彩画のような柔らかいタッチに変換 |

> **AIスタイル変換の仕組み**: Stability AI Style Transfer (`us.stability.stable-style-transfer-v1:0`) を使用。
> 事前に用意した「参照スタイル画像」と撮影した写真を入力すると、
> 写真の構図・人物はそのままに画風だけをAIが変換する。
> `style_strength`, `composition_fidelity`, `change_strength` パラメータで変換の度合いを調整可能。
>
> **コスト**: ~$0.03-0.06/枚 × 4枚 = ~$0.12-0.24/セッション
>
> **注意**: AIフィルター選択時は処理時間が長くなることをユーザーに事前に表示する。

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目 | 要件（高速化後） | 旧要件 |
|------|-----------------|--------|
| 4枚撮影〜印刷完了（簡易フィルター） | **12秒以内**（撮影時間除く） | 20秒以内 |
| 4枚撮影〜印刷完了（AIスタイル） | **30秒以内**（撮影時間除く） | 40秒以内 |
| 画像アップロード（4枚） | **2秒以内**（WiFi環境） | 5秒以内 |
| 画像処理（簡易フィルター） | **6秒以内** | 5秒以内 |
| 画像処理（AIスタイル変換） | **24秒以内**（4枚並列） | 20秒以内 |
| 印刷所要時間 | **5-7秒** | 10秒以内 |
| 同時接続ユーザー | 50人以上 | 同左 |

#### 高速化施策一覧

**1. アップロード高速化（5秒 → 1-2秒）**

| 施策 | 効果 |
|------|------|
| S3 Transfer Acceleration | CloudFrontエッジ経由でアップロード高速化 |
| クライアント側画像圧縮 | Canvas APIで1280pxリサイズ + JPEG品質80%に圧縮してからアップ |
| 4枚同時アップロード | `Promise.all` で4枚を完全並列PUT |

**2. Lambda実行高速化**

| 施策 | 効果 |
|------|------|
| Provisioned Concurrency | コールドスタート（~3秒）を完全排除 |
| メモリ増量（10GB） | メモリに比例してCPUも増加。画像処理が2-3倍速に |
| ARM64 (Graviton2) | x86比で最大20%高速 + 20%安価 |

**3. パイプライン超並列化（最大の改善効果）**

```
Step 1 (並列):
├─ 顔検出(Rekognition) ─┐
├─ フィルター適用(4枚並列) ─┤
│                         │
Step 2 (並列):            ▼
├─ コラージュ生成 ──────────┐
├─ キャプション生成(Bedrock) ← フィルター済み画像から即座に開始
│                           │
Step 3 (並列):              ▼
├─ 感情分析(Comprehend) ← キャプション完了後即座に
├─ ディザリング+フレーム ← コラージュ完了後即座に開始、感情結果を待って最終合成
│                           │
├─ 印刷ジョブ送信 ←──────── 完了と同時
```

**4. Step Functions Express Workflows**

| 施策 | 効果 |
|------|------|
| Standard → Express Workflows | 各ステート遷移の~1秒オーバーヘッドを排除。6ステップで~6秒短縮 |

**5. 印刷高速化（10秒 → 5-7秒）**

| 施策 | 効果 |
|------|------|
| 先行ダウンロード | ディザリング完了をIoT MQTTで即通知→S3ダウンロード先行開始 |
| ESC/POSラスター事前変換 | Lambda側で印刷コマンドに変換済みバイナリを生成 |
| プリンター速度最大化 | MUNBYN 230mm/sモード活用 |

### 4.2 可用性

| 項目 | 要件 |
|------|------|
| サービス稼働時間 | イベント開催中は100%（数時間の短期運用） |
| プリンター障害 | エラー通知 + ユーザーへのフォールバック表示 |

### 4.3 セキュリティ

| 項目 | 要件 |
|------|------|
| 認証 | なし（匿名利用） |
| 画像保存 | イベント終了後に一括削除可能にする（TTL設定） |
| Presigned URL | 有効期限を短く設定（5分） |
| HTTPS | 全通信をHTTPS化（カメラAPI利用に必須） |

### 4.4 対応ブラウザ・端末

| 項目 | 要件 |
|------|------|
| iOS Safari | 必須対応 |
| Android Chrome | 必須対応 |
| デスクトップ | QRコード表示・ダッシュボード用のみ |

### 9.2 フィルター処理の実装方針

#### 簡易フィルター（Lambda + sharp）

| フィルター | sharp実装 | 処理時間 |
|-----------|-----------|---------|
| ナチュラル | 処理なし（元画像のまま） | 即座 |
| 美肌 | `blur` で平滑化 → 元画像と `blend` でブレンド | ~1秒 |
| 明るさ補正 | `modulate` + `linear` で明るさ・コントラスト補正 | ~1秒 |
| モノクロ | `greyscale()` | ~1秒 |
| セピア | グレースケール化 → セピアカラーマップ適用 | ~1秒 |

#### AIスタイル変換（Bedrock Stability AI Style Transfer）

| スタイル | 参照画像 | パラメータ目安 |
|---------|---------|--------------|
| アニメ風 | アニメ調のイラスト参照画像を事前用意 | `style_strength: 0.6`, `composition_fidelity: 0.8` |
| ポップアート風 | ポップアートの参照画像を事前用意 | `style_strength: 0.7`, `composition_fidelity: 0.7` |
| 水彩画風 | 水彩画の参照画像を事前用意 | `style_strength: 0.5`, `composition_fidelity: 0.8` |

**処理フロー:**
1. 4枚の写真に対してBedrock APIを**並列呼び出し**
2. 各写真 + 参照スタイル画像 → Style Transfer API → 変換済み画像
3. 並列処理により、4枚でも壁時計時間は1枚分（~10-20秒）

**コスト**: ~$0.03-0.06/枚 × 4枚 = ~$0.12-0.24/セッション

### 10.2 ローカル印刷サービス

MacBook上で動作する印刷制御サービス。IoT Core MQTTを主通信方式とし、SQSポーリングをフォールバックとする。

**処理フロー（IoT Core MQTT + WebUSB方式 - 推奨）:**
1. PC ブラウザが AWS IoT Core に MQTT over WSS で接続
2. トピック `receipt-purikura/print/{sessionId}` で印刷ジョブを受信
3. S3から印刷用コラージュ画像をダウンロード
4. WebUSB API で ESC/POS ラスターコマンドに変換・USB プリンターに送信
5. 自動カット命令を送信
6. トピック `receipt-purikura/print/{sessionId}/status` で印刷完了を通知

**処理フロー（SQS + python-escpos フォールバック方式）:**
1. MacBook 上で Docker コンテナとして印刷サービスを起動
2. SQS キューをロングポーリング（20秒間隔）
3. python-escpos で印刷

**エラーハンドリング:**
- プリンター接続エラー → リトライ（3回）→ SQSにメッセージを戻す
- 紙切れ → MacBookに通知を表示
- IoT Core接続失敗 → SQSポーリングに自動フォールバック

## 13. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| AIスタイル変換の品質が不安定 | 中 | 参照スタイル画像とパラメータを事前に十分チューニング。最悪簡易フィルターのみで運用 |
| AIスタイル変換のレイテンシ | 中 | 4枚並列処理で壁時計時間を短縮。処理中UIで待ち時間を演出 |
| Stability AI Style Transferのコスト | 低 | ~$0.24/セッション。ハッカソン期間中は許容範囲 |
| プリンターの画質が粗い | 中 | ディザリングアルゴリズムの事前検証で最適化 |
| 4枚撮影のUXが悪い | 中 | カウントダウンUI・撮り直し機能で体験を向上 |
| **iOS SafariのWebRTC + カメラ挙動** | **高** | getUserMedia + WebRTC の機種差が大きい。**Day 2-3で実機検証必須**。adapter.jsポリフィル使用 |
| **WebRTC接続が会場ネットワークで確立不可** | **高** | フォールバック: WebSocket経由のフレーム送信（~500ms遅延） |
| **WebSocket構築の工数オーバー** | **高** | 詰まったら即ポーリング（5秒間隔GET）に切り替える |
| **19サービス全部の実装が間に合わない** | **高** | 段階的リリース（Stage 1-6）に従い、フォールバック戦略で各機能を軽量化 |
| **IaC (CDK) 構築の工数オーバー** | **高** | 手動構築で進め、余裕があればコード化 |
| ブラウザAutoplay Policy | 中 | PC側で「開始」ボタンクリックによるユーザーインタラクション取得 |
| IoT Core MQTT接続の安定性 | 中 | SQSポーリングをフォールバックとして常に準備 |
| MediaPipe ARのパフォーマンス | 中 | PC側のGPU性能に依存。重い場合はAR無しに切り替え |
| Polly音声生成のレイテンシ | 低 | カウントダウン音声は事前生成してS3にキャッシュ |
| 会場WiFiの速度不足 | 中 | 画像を圧縮してアップロード。モバイルテザリングも準備 |
| Lambda コールドスタート | 低 | Provisioned Concurrencyの設定 |
| 50人同時接続時の遅延 | 中 | SQS + Step Functionsで非同期処理。印刷は直列だが処理は並列 |
| プリンター紙切れ | 低 | 予備ロール紙を十分に準備 |
| 4枚アップロードの通信量 | 中 | 画像をリサイズ・圧縮（最大1280px、JPEG品質80%） |
| AWSコスト増加 | 中 | 追加サービス（Polly, Transcribe, Comprehend等）はセッション単価が低い。ハッカソン期間中は許容 |

## 17. 用語集

| 用語 | 説明 |
|------|------|
| ディザリング | カラーやグレースケールの画像を白黒2値（白/黒のドットのみ）に変換する技術。ドットの密度で濃淡を表現する |
| コラージュ | 複数の写真を1枚の画像にレイアウトして合成すること。本プロジェクトでは4枚を2x2グリッドに配置する |
| ESC/POS | レシートプリンターの標準制御コマンド体系。ほぼ全てのレシートプリンターが対応 |
| Presigned URL | 一時的にS3への直接アップロードを許可する署名付きURL。サーバーを経由せず直接S3にアップロードできる |
| Floyd-Steinberg | 最も一般的なディザリングアルゴリズム。誤差を周囲のピクセルに拡散させて自然な階調を表現する |
| WebSocket | サーバーからクライアントにリアルタイムでデータを送信できる双方向通信プロトコル |
| WebRTC | ブラウザ間でリアルタイムの音声・映像・データ通信を行うP2P技術。本プロジェクトではスマホ→PCの映像ストリーミングに使用 |
| SDP | Session Description Protocol。WebRTC接続確立時に交換されるメディア情報の記述プロトコル |
| STUN/TURN | WebRTCのP2P接続確立を補助するサーバー。STUNはNAT越え、TURNはリレーサーバー |
| MediaPipe | Googleが提供するMLフレームワーク。Face Meshで顔の468点のランドマークをリアルタイム検出し、ARフィルターを実現 |
| MQTT | IoTデバイス向けの軽量メッセージングプロトコル。AWS IoT Coreで使用。パブリッシュ/サブスクライブモデル |
| EventBridge | AWSのサーバーレスイベントバス。イベント駆動アーキテクチャの中核 |
| AppSync | AWSのマネージドGraphQLサービス。サブスクリプションでリアルタイムデータ同期が可能 |
| DynamoDB Streams | DynamoDBテーブルへの変更をリアルタイムでキャプチャするイベントソーシング機能 |
| Lambda@Edge | CloudFrontのエッジロケーションでLambda関数を実行する機能。リクエスト/レスポンスをエッジで加工 |
| Amazon Polly | テキストをリアルな音声に変換するAI音声合成サービス |
| Amazon Transcribe | 音声をテキストに変換する自動音声認識（ASR）サービス |
| Amazon Comprehend | テキストの感情分析・エンティティ抽出等を行う自然言語処理サービス |
| CloudWatch Synthetics | カナリアスクリプトによるWebアプリケーションの外形監視サービス |
| ServiceLens | CloudWatch + X-Rayを統合し、マイクロサービスマップを可視化するサービス |
| Step Functions | AWSのワークフローオーケストレーションサービス。複数のLambda関数を順番/並列に実行する |
| SQS | AWSのメッセージキューサービス。非同期でジョブを受け渡す仕組み |
| sharp | Node.jsの高速画像処理ライブラリ（libvips基盤）。フィルター適用、リサイズ、合成などの画像操作ができる |
| AWS CDK | AWSのInfrastructure as Code（IaC）フレームワーク。TypeScriptやPythonでインフラを定義 |
